Req & Res
O ciclo de vida

Entender req e res é entender como o Express funciona por dentro. Toda requisição que chega ao seu servidor passa por esses dois objetos — eles carregam tudo que você precisa para ler, processar e responder.

Nível 01 — Básico Teoria + Código Essencial

01 O ciclo HTTP — a base de tudo

Antes de entrar no Express, você precisa entender o protocolo que ele usa: o HTTP (HyperText Transfer Protocol). Todo servidor web — Express, Django, Rails, Spring — funciona em cima dessa mesma lógica.

O HTTP é um protocolo de requisição e resposta. Isso significa que a comunicação sempre segue esse padrão: alguém envia uma requisição, o servidor processa e devolve uma resposta. Simples assim.

Ciclo de uma requisição
🌐 Cliente
(Browser / App)
──── req ────▶
Express
processa
◀──── res ────
🌐 Cliente
recebe resposta
req → contém: método, URL, headers, body, params
res → contém: status, headers, body da resposta

Quando o Express recebe uma requisição HTTP, ele cria automaticamente dois objetos e os passa para a sua função de rota:

Estrutura básica de uma rota JavaScript 
app.get('/usuarios', (req, res) => {
  // req = Request  → o que o cliente ENVIOU para o servidor
  // res = Response → o que o servidor vai DEVOLVER ao cliente

  res.send('Hello!')
})
// Conceito chave req e res são objetos criados pelo Express a partir da requisição HTTP. Eles extendem os objetos nativos do Node.js (http.IncomingMessage e http.ServerResponse) adicionando dezenas de propriedades e métodos úteis.

02 O objeto req — a requisição

O objeto req (request) representa tudo que o cliente enviou ao servidor. Pense nele como uma "caixa" que contém todas as informações da requisição: de onde veio, o que quer, quais dados trouxe.

req.method

Indica o verbo HTTP usado na requisição. É sempre uma string em maiúsculo.

req.methodJavaScript 
app.all('/teste', (req, res) => {
  console.log(req.method)
  // GET, POST, PUT, PATCH, DELETE...
})

req.url e req.path

req.url retorna a URL completa incluindo query string. req.path retorna apenas o caminho, sem parâmetros.

req.url vs req.pathJavaScript
// Requisição para: GET /produtos?categoria=eletronicos&pagina=2

app.get('/produtos', (req, res) => {
  console.log(req.url)   // '/produtos?categoria=eletronicos&pagina=2'
  console.log(req.path)  // '/produtos'
})

req.params

Contém os parâmetros de rota — valores dinâmicos definidos com : na URL. São parte da estrutura da rota, não opcionais.

req.paramsJavaScript 
// Rota com parâmetros dinâmicos
app.get('/usuarios/:id/posts/:postId', (req, res) => {
  console.log(req.params)
  // { id: '42', postId: '7' }

  const { id, postId } = req.params
  res.json({ usuarioId: id, postId: postId })
})

// GET /usuarios/42/posts/7
// ↑ id = '42'   postId = '7'
// Atenção Os valores de req.params são sempre strings, mesmo que pareçam números. Se precisar de um número, converta: Number(req.params.id) ou parseInt(req.params.id).

req.query

Contém os parâmetros de query string — os valores que vêm após o ? na URL. São opcionais e usados para filtros, paginação, ordenação.

req.queryJavaScript 
// GET /produtos?categoria=eletronicos&pagina=2&limite=10

app.get('/produtos', (req, res) => {
  console.log(req.query)
  // { categoria: 'eletronicos', pagina: '2', limite: '10' }

  const { categoria, pagina = '1', limite = '10' } = req.query

  res.json({
    categoria,
    pagina: Number(pagina),
    limite: Number(limite)
  })
})

req.body

Contém os dados enviados no corpo da requisição. Geralmente usado com POST, PUT e PATCH. Para acessar o body, o Express precisa de um middleware de parsing.

req.body com middlewareJavaScript
const express = require('express')
const app = express()

// Middleware necessário para ler req.body em JSON
app.use(express.json())

app.post('/usuarios', (req, res) => {
  console.log(req.body)
  // { nome: 'João', email: 'joao@email.com', idade: 28 }

  const { nome, email } = req.body
  res.status(201).json({ mensagem: `Usuário ${nome} criado!` })
})

req.headers

Contém todos os headers HTTP enviados pelo cliente. Úteis para autenticação (Authorization), tipo de conteúdo (Content-Type), informações do cliente (User-Agent), etc.

req.headersJavaScript 
app.get('/perfil', (req, res) => {
  // Headers estão sempre em lowercase
  const token = req.headers['authorization']
  const contentType = req.headers['content-type']
  const userAgent = req.headers['user-agent']

  // Atalho para um header específico
  const auth = req.get('Authorization') // case-insensitive

  res.json({ token, userAgent })
})

Outras propriedades úteis do req

Propriedade Tipo Descrição
req.ip String IP do cliente que fez a requisição
req.hostname String Hostname do servidor (ex: localhost)
req.protocol String http ou https
req.secure Boolean True se a conexão for HTTPS
req.cookies Object Cookies (requer middleware cookie-parser)
req.originalUrl String URL original antes de qualquer reescrita
req.xhr Boolean True se a req veio via XMLHttpRequest (AJAX)

03 O objeto res — a resposta

O objeto res (response) é o que você usa para construir e enviar a resposta de volta ao cliente. Você decide o status code, os headers e o corpo da resposta.

// Regra fundamental Você deve sempre encerrar a requisição enviando uma resposta. Se não enviar, o cliente ficará esperando para sempre até o timeout. Use res.send(), res.json(), res.redirect() ou res.end().

res.send()

O método mais genérico. Envia uma resposta de qualquer tipo — string, HTML, Buffer, objeto. O Content-Type é definido automaticamente pelo tipo do dado.

res.send()JavaScript 
// Enviando uma string
res.send('Olá mundo!')
// Content-Type: text/html

// Enviando HTML
res.send('<h1>Bem vindo!</h1>')

// Enviando um Buffer
res.send(Buffer.from('hello'))
// Content-Type: application/octet-stream

res.json()

Envia uma resposta JSON. Define automaticamente o Content-Type: application/json e serializa o objeto. Preferido para APIs REST.

res.json()JavaScript 
// Enviando um objeto como JSON
res.json({ nome: 'João', idade: 28 })

// Enviando um array
res.json([{ id: 1 }, { id: 2 }])

// Combinando com status
res.status(201).json({
  sucesso: true,
  dados: { id: 1, nome: 'João' }
})

res.status()

Define o HTTP status code da resposta. Retorna o próprio res, permitindo encadeamento (chaining). Sempre use o status correto para comunicar o resultado da operação.

res.status()JavaScript 
// Status sem corpo
res.status(204).end()   // No Content (delete, por exemplo)

// Status com JSON (o padrão)
res.status(200).json({ ok: true })
res.status(201).json({ id: 1 })        // Created
res.status(400).json({ erro: 'Dados inválidos' })
res.status(401).json({ erro: 'Não autorizado' })
res.status(404).json({ erro: 'Não encontrado' })
res.status(500).json({ erro: 'Erro interno' })

res.redirect()

Redireciona o cliente para outra URL. O status padrão é 302 (Found/Temporary), mas pode ser 301 (Permanent).

res.redirect()JavaScript
// Redirect temporário (302 padrão)
res.redirect('/nova-url')

// Redirect permanente (301)
res.redirect(301, '/nova-url-permanente')

// Redirect para URL externa
res.redirect('https://google.com')

// Redirect relativo (volta uma pasta)
res.redirect('..')

res.sendFile()

Envia um arquivo como resposta. O caminho deve ser absoluto ou relativo ao root definido nas opções.

res.sendFile()JavaScript
const path = require('path')

app.get('/download', (req, res) => {
  const filePath = path.join(__dirname, 'files', 'relatorio.pdf')
  res.sendFile(filePath)
})

res.set() e res.get()

Para definir ou ler headers na resposta manualmente.

Manipulando headers da respostaJavaScript
app.get('/dados', (req, res) => {
  // Definindo um header
  res.set('X-Custom-Header', 'meu-valor')
  res.set('Cache-Control', 'no-cache')

  // Definindo múltiplos de uma vez
  res.set({
    'Content-Type': 'application/json',
    'X-Powered-By': 'Express'
  })

  // Lendo um header da resposta
  console.log(res.get('Content-Type'))

  res.json({ ok: true })
})

04 Status Codes — comunicando o resultado

Os status codes HTTP são a linguagem que o servidor usa para dizer ao cliente o que aconteceu com a requisição. Usar o status correto é uma questão de semântica e boa prática de API.

200
OK
Sucesso genérico. Usado em GET bem-sucedido.
201
Created
Recurso criado com sucesso. Usar em POST.
204
No Content
Sucesso sem corpo. Ideal para DELETE.
301
Moved Perm.
Redirect permanente. Browsers guardam em cache.
302
Found
Redirect temporário. O padrão do res.redirect().
400
Bad Request
Dados inválidos ou mal formatados enviados.
401
Unauthorized
Não autenticado. Precisa fazer login.
403
Forbidden
Autenticado, mas sem permissão.
404
Not Found
Recurso não encontrado no servidor.
422
Unprocessable
Dados válidos mas falham nas regras de negócio.
500
Server Error
Erro interno. Nunca esperado em produção.
503
Unavailable
Servidor sobrecarregado ou em manutenção.
// Dica de ouro 4xx = erro do lado do cliente (ele fez algo errado) → nunca logue como erro crítico.
5xx = erro do lado do servidor (você fez algo errado) → sempre monitore e alerte.

05 Headers — metadados da comunicação

Headers são pares chave-valor que carregam metadados sobre a requisição ou resposta. Eles não fazem parte do conteúdo em si, mas informam ao servidor/cliente como interpretar aquele conteúdo.

Header Direção Descrição
Content-Type Req + Res Tipo do conteúdo: application/json, text/html, multipart/form-data
Authorization Request Token de autenticação: Bearer eyJhbGc...
Accept Request Quais tipos de resposta o cliente aceita
Cache-Control Res Instruções de cache: no-cache, max-age=3600
Location Response URL de redirecionamento após 201 ou 3xx
User-Agent Request Identifica o cliente: browser, app, curl...
Cookie / Set-Cookie Req / Res Envio e definição de cookies

06 Body — o conteúdo da mensagem

O body (corpo) é onde os dados reais são transportados. Nem todas as requisições têm body — GET e DELETE geralmente não têm. POST, PUT e PATCH quase sempre têm.

Tipos de body mais comuns

Content-Type Quando usar Middleware necessário
application/json APIs REST — o padrão atual express.json()
application/x-www-form-urlencoded Formulários HTML tradicionais express.urlencoded()
multipart/form-data Upload de arquivos multer (pacote externo)
text/plain Texto simples express.text()
// Erro comum Se você não adicionar app.use(express.json()), o req.body sempre será undefined — e você vai perder horas debugando. Sempre configure o middleware de parsing antes das suas rotas.

07 Código completo — CRUD real

Veja tudo que aprendemos em um exemplo real: uma API de usuários com as 4 operações básicas, usando req e res corretamente em cada cenário.

API de Usuários — CRUD completoJavaScript
const express = require('express')
const app = express()

// Middleware para parsear JSON no body
app.use(express.json())

// Banco de dados em memória (simulação)
let usuarios = [
  { id: 1, nome: 'Ana', email: 'ana@email.com' },
  { id: 2, nome: 'Bruno', email: 'bruno@email.com' },
]

// ─────────────────────────────────────────
// GET /usuarios — Lista todos os usuários
// Usa: req.query para filtros opcionais
// ─────────────────────────────────────────
app.get('/usuarios', (req, res) => {
  const { nome } = req.query

  let resultado = usuarios
  if (nome) {
    resultado = usuarios.filter(u =>
      u.nome.toLowerCase().includes(nome.toLowerCase())
    )
  }

  res.status(200).json({
    total: resultado.length,
    dados: resultado
  })
})

// ─────────────────────────────────────────
// GET /usuarios/:id — Busca um usuário
// Usa: req.params.id
// ─────────────────────────────────────────
app.get('/usuarios/:id', (req, res) => {
  const id = Number(req.params.id)
  const usuario = usuarios.find(u => u.id === id)

  if (!usuario) {
    return res.status(404).json({ erro: 'Usuário não encontrado' })
  }

  res.status(200).json(usuario)
})

// ─────────────────────────────────────────
// POST /usuarios — Cria um usuário
// Usa: req.body (nome, email)
// ─────────────────────────────────────────
app.post('/usuarios', (req, res) => {
  const { nome, email } = req.body

  if (!nome || !email) {
    return res.status(400).json({ erro: 'Nome e email são obrigatórios' })
  }

  const novoUsuario = {
    id: usuarios.length + 1,
    nome,
    email
  }

  usuarios.push(novoUsuario)

  res.status(201).json(novoUsuario)
})

// ─────────────────────────────────────────
// PUT /usuarios/:id — Atualiza um usuário
// Usa: req.params.id + req.body
// ─────────────────────────────────────────
app.put('/usuarios/:id', (req, res) => {
  const id = Number(req.params.id)
  const index = usuarios.findIndex(u => u.id === id)

  if (index === -1) {
    return res.status(404).json({ erro: 'Usuário não encontrado' })
  }

  usuarios[index] = { ...usuarios[index], ...req.body, id }

  res.status(200).json(usuarios[index])
})

// ─────────────────────────────────────────
// DELETE /usuarios/:id — Remove um usuário
// Usa: req.params.id — Resposta: 204 sem body
// ─────────────────────────────────────────
app.delete('/usuarios/:id', (req, res) => {
  const id = Number(req.params.id)
  const existe = usuarios.some(u => u.id === id)

  if (!existe) {
    return res.status(404).json({ erro: 'Usuário não encontrado' })
  }

  usuarios = usuarios.filter(u => u.id !== id)

  res.status(204).end() // Sem corpo na resposta
})

app.listen(3000, () => {
  console.log('Servidor rodando em http://localhost:3000')
})

08 Resumo — o que fixar

Objeto Propriedade / Método Para quê serve
req req.method Verbo HTTP da requisição
req.params Parâmetros da URL (/rota/:id)
req.query Query string (?chave=valor)
req.body Dados do corpo (precisa de middleware)
req.headers Headers enviados pelo cliente
req.get(name) Lê um header específico (case-insensitive)
req.ip / req.path IP do cliente e caminho da URL
res res.status(code) Define o status HTTP (retorna res para chaining)
res.json(obj) Envia resposta JSON (padrão para APIs)
res.send(data) Envia resposta genérica (string, HTML, Buffer)
res.redirect(url) Redireciona o cliente
res.sendFile(path) Envia um arquivo como resposta
res.set(key, val) Define headers na resposta
res.end() Finaliza a resposta sem corpo
