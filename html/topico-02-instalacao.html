Estrutura Inicial

Do zero ao primeiro servidor funcionando. Aqui você aprende a criar um projeto Node.js do zero, entender cada campo do package.json, instalar o Express e montar uma estrutura de pastas que vai escalar junto com o projeto.

Nível 01 — Básico Setup Configuração

01Pré-requisitos

Para usar o Express você precisa ter o Node.js instalado. O Express é um framework que roda em cima do Node — ele não funciona no browser.

bash 
$ node --version
v20.11.0

$ npm --version
10.2.4

Se o comando falhar, baixe o Node.js em nodejs.org — o npm vem junto automaticamente. Use sempre uma versão LTS (Long Term Support) em projetos profissionais.

Ferramenta O que é Versão recomendada
Node.js Runtime JavaScript no servidor 20.x LTS ou superior
npm Gerenciador de pacotes (vem com Node) Qualquer recente
VS Code Editor recomendado Qualquer versão
// npm vs yarn vs pnpm Todos são gerenciadores de pacotes e fazem a mesma coisa. npm é o padrão e já vem com o Node — use ele. Se você já usa yarn ou pnpm, pode substituir os comandos tranquilamente (ex: npm install → yarn add).

02Criando o projeto com npm init

Todo projeto Node.js começa com o comando npm init. Ele cria o arquivo package.json, que é o coração do projeto — registra nome, versão, dependências e scripts.

1
Crie e entre na pasta do projeto
bash 
$ mkdir minha-api
$ cd minha-api
2
Inicialize o projeto
bash 
$ npm init -y
Wrote to /minha-api/package.json:
{
  "name": "minha-api",
  "version": "1.0.0",
  ...
}
A flag -y responde "sim" para todas as perguntas e cria o arquivo com valores padrão. Sem ela, o npm faz perguntas interativas.
3
Abra no editor
bash 
$ code .

03Entendendo o package.json

O package.json é o manifesto do projeto. Ele descreve tudo sobre o seu projeto e é o arquivo mais importante depois do seu código. Veja o que cada campo significa:

package.json completoJSON
{
  "name": "minha-api",
  "version": "1.0.0",
  "description": "API REST com Express",
  "main": "src/server.js",
  "scripts": {
    "start": "node src/server.js",
    "dev": "nodemon src/server.js",
    "test": "jest"
  },
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "nodemon": "^3.0.1"
  },
  "engines": {
    "node": ">=18.0.0"
  },
  "author": "Seu Nome",
  "license": "MIT"
}
"name"
Nome do projeto. Deve ser único se publicar no npm. Use letras minúsculas e hífens, sem espaços.
"version"
Segue o padrão Semantic Versioning: MAJOR.MINOR.PATCH. Ex: 2.1.3 significa versão maior 2, feature menor 1, correção de bug 3.
"main"
Arquivo de entrada do projeto — o ponto de partida quando alguém importa seu pacote. Para uma API, aponte para seu arquivo principal.
"scripts"
Atalhos de terminal executados com npm run <nome>. O script start é especial: roda com npm start sem o "run".
"dependencies"
Pacotes necessários em produção. São instalados com npm install <pacote>. Vão para o servidor quando você fizer deploy.
"devDependencies"
Pacotes usados somente em desenvolvimento — testes, linters, ferramentas de build. Instalados com npm install -D <pacote>. Não vão para produção.
// O que é o ^ no número de versão? "express": "^4.18.2" — o ^ (caret) significa "aceite atualizações de versão minor e patch, mas não de major". Então ^4.18.2 aceita 4.18.3, 4.19.0, mas não 5.0.0. Isso protege de breaking changes acidentais.

04Instalando o Express

bash 
$ npm install express

added 64 packages in 2s
found 0 vulnerabilities

Após rodar o comando, três coisas acontecem no seu projeto:

📁 minha-api/
├── 📁 node_modules/ ← criada agora (não toque aqui)
├── 📄 package.json ← atualizado com "express" em dependencies
└── 📄 package-lock.json ← criado agora (versões exatas travadas)

O que é a pasta node_modules?

É onde o npm salva fisicamente todos os pacotes instalados. A pasta do Express fica em node_modules/express/. Ela pode ter centenas de subpastas porque cada pacote tem suas próprias dependências.

// NUNCA suba node_modules para o Git A pasta node_modules pode ter centenas de MB. Ela é gerada automaticamente com npm install a partir do package.json — não precisa ser versionada. Adicione sempre ao .gitignore.

O que é o package-lock.json?

Enquanto o package.json define intervalos de versão (^4.18.2), o package-lock.json trava as versões exatas de todos os pacotes instalados. Isso garante que todos no time, e o servidor em produção, instalem exatamente os mesmos pacotes.

// Commits e package-lock.json Sempre commite o package-lock.json no Git. Ele é o que garante reprodutibilidade — qualquer pessoa que clonar o repositório e rodar npm install terá o ambiente idêntico ao seu.

05O primeiro servidor Express

Com o Express instalado, crie o arquivo principal e suba o servidor:

server.js — o mínimo funcionalJavaScript
// 1. Importa o Express
const express = require('express')

// 2. Cria uma instância da aplicação
const app = express()

// 3. Define a porta
const PORT = 3000

// 4. Cria uma rota
app.get('/', (req, res) => {
  res.send('Servidor Express funcionando! 🚀')
})

// 5. Inicia o servidor
app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`)
})
bash 
$ node server.js
Servidor rodando em http://localhost:3000

Abra http://localhost:3000 no browser ou teste com curl:

bash 
$ curl http://localhost:3000
Servidor Express funcionando! 🚀

Linha por linha — o que cada parte faz

Linha O que faz
require('express') Carrega o módulo Express instalado na node_modules
express() Cria uma instância da aplicação — o objeto app é tudo
app.get('/', ...) Registra um handler para GET na rota raiz /
res.send() Envia a resposta e encerra a conexão
app.listen(PORT, cb) Inicia o servidor HTTP na porta indicada

06Estrutura de pastas recomendada

O Express não impõe nenhuma estrutura — você pode colocar tudo em um arquivo só. Mas para projetos que crescem, uma boa organização desde o começo evita dor de cabeça.

Estrutura simples (projetos pequenos)

📁 minha-api/
├── 📁 src/
│  ├── 📄 server.js ← ponto de entrada, inicia o app
│  ├── 📄 app.js ← instância do Express + middlewares
│  ├── 📁 routes/ ← arquivos de rota por recurso
│  │  └── 📄 usuarios.js
│  └── 📁 controllers/ ← lógica de negócio
│     └── 📄 usuariosController.js
├── 📄 .env ← variáveis de ambiente (nunca no Git)
├── 📄 .env.example ← template do .env (vai no Git)
├── 📄 .gitignore
├── 📄 package.json
└── 📄 README.md

Separando app.js de server.js

Uma convenção muito comum e útil é separar a criação do app da inicialização do servidor. Isso facilita muito os testes automatizados — você pode importar o app sem subir o servidor.

src/app.js — configuração da aplicaçãoJavaScript
const express = require('express')
const app = express()

// Middlewares globais
app.use(express.json())
app.use(express.urlencoded({ extended: true }))

// Rotas
app.get('/', (req, res) => {
  res.json({ status: 'ok', versao: '1.0.0' })
})

// Exporta o app (sem chamar .listen)
module.exports = app
src/server.js — inicialização do servidorJavaScript
const app = require('./app')

const PORT = process.env.PORT || 3000

app.listen(PORT, () => {
  console.log(`🚀 Servidor rodando em http://localhost:${PORT}`)
})
// Por que separar? Nos testes com Jest + Supertest (Tópico 19), você importa apenas o app.js sem precisar subir o servidor real. Isso torna os testes mais rápidos, isolados e sem conflito de porta.

07Scripts npm

Os scripts no package.json são atalhos de terminal. Em vez de digitar node src/server.js toda vez, você define uma vez e roda com npm run dev.

Scripts no package.jsonJSON
{
  "scripts": {
    "start": "node src/server.js",       // produção
    "dev":   "nodemon src/server.js",    // desenvolvimento
    "test":  "jest --coverage",           // testes
    "lint":  "eslint src/"              // verificação de código
  }
}
bash 
$ npm start         # roda em produção
$ npm run dev       # roda em desenvolvimento (com nodemon)
$ npm test          # roda os testes
$ npm run lint      # verifica qualidade do código
// start e test são especiais Os scripts start e test são reservados — você roda com npm start e npm test (sem o "run"). Todos os outros precisam do npm run antes.

08Nodemon — reload automático

Por padrão, o Node.js não reinicia o servidor quando você salva um arquivo. O nodemon resolve isso: ele monitora os arquivos e reinicia automaticamente toda vez que detecta uma mudança.

bash 
$ npm install -D nodemon
added 34 packages in 1s

A flag -D instala como devDependency — o nodemon só é necessário em desenvolvimento, não em produção.

bash — nodemon em ação 
$ npm run dev

[nodemon] 3.0.1
[nodemon] watching: *.*
[nodemon] starting `node src/server.js`
Servidor rodando em http://localhost:3000

# Você salva um arquivo...

[nodemon] restarting due to changes...
[nodemon] starting `node src/server.js`
Servidor rodando em http://localhost:3000

Configurando o nodemon (opcional)

Você pode criar um arquivo nodemon.json na raiz para configurar quais arquivos monitorar, quais ignorar e outras opções:

nodemon.jsonJSON
{
  "watch": ["src"],
  "ext": "js,json",
  "ignore": ["src/**/*.test.js", "node_modules"],
  "delay": 500
}

Variáveis de ambiente com dotenv

Nunca coloque senhas, chaves de API ou strings de conexão diretamente no código. Use variáveis de ambiente — elas ficam num arquivo .env que não vai para o Git.

bash 
$ npm install dotenv
.env — variáveis do projetotext
PORT=3000
NODE_ENV=development
DB_URL=mongodb://localhost:27017/minha-api
JWT_SECRET=sua-chave-secreta-aqui
.env.example — template para o timetext
# Copie este arquivo para .env e preencha os valores
PORT=3000
NODE_ENV=development
DB_URL=
JWT_SECRET=
src/server.js — carregando o .envJavaScript
// Deve ser a PRIMEIRA linha — antes de qualquer import
require('dotenv').config()

const app = require('./app')

// Agora process.env tem todas as variáveis do .env
const PORT = process.env.PORT || 3000
const ENV  = process.env.NODE_ENV || 'development'

app.listen(PORT, () => {
  console.log(`🚀 [${ENV}] http://localhost:${PORT}`)
})

❌ Nunca faça

  • Colocar senhas no código
  • Commitar o arquivo .env
  • Usar a mesma chave em dev e prod
  • Hardcodar URLs de banco

✅ Boas práticas

  • Tudo sensível fica no .env
  • .env está no .gitignore
  • .env.example está no Git
  • Valores padrão seguros no código
